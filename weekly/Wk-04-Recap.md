# Week 4 Recap: Anomaly Detection Benchmark

This week focused on turning the Week 3 synthetic Rover telemetry work into a more complete anomaly-detection benchmark. In Week 3, the main goal was to generate and understand the telemetry dataset. This week, the goal shifted toward asking a more operational question: if a fleet-health system receives this kind of telemetry, which anomaly detection method is actually useful under different deployment constraints?

The first major task was building the Week 4 benchmark notebook. I used the 5-unit, 1-day Rover telemetry sample as the default input so the analysis would stay reproducible and realistic to run on a local machine. Inside the notebook, I injected five controlled fault patterns: motor stall, gradual sensor drift, accelerated battery degradation, GPS jitter with spoofing-like behavior, and intermittent software hang. I paid special attention to leakage control because it would be easy to accidentally make the benchmark look better than it really is. The notebook splits the data before injecting faults, keeps training windows clean, fits scalers only on the training set, and evaluates each method on held-out fault windows.

The second major task was benchmarking four anomaly detection methods: Isolation Forest, One-Class SVM, Local Outlier Factor, and an LSTM autoencoder. I compared them across 10 seeded train/test splits and reported precision, recall, F1, and AUROC by fault type. I also added paired Wilcoxon signed-rank tests on AUROC differences, which made the evaluation feel closer to the statistical comparison discipline I have used before in finance and backtesting. Instead of relying on one lucky run, the benchmark now gives a more defensible view of which method is more stable across repeated splits.

A key learning this week was that anomaly detection is not just about the best overall score. Different operating regimes lead to different recommendations. For a latency-constrained setting, a simpler method such as Isolation Forest is easier to justify as a first-pass screen because it is faster and more operationally straightforward. For an accuracy-maximizing setting, LOF looked stronger in the benchmark results, especially when the goal is to catch more subtle deviations. The LSTM autoencoder was useful conceptually, but it also introduced more complexity, runtime cost, and tuning uncertainty than I would want to recommend at this stage.

I also prepared the one-page method recommendation memo and the mid-point summary deck for supervisor review. The memo forced me to translate model results into deployment guidance, including explicit assumptions and limitations. The deck summarized Phases A and B, connecting the earlier physical AI landscape and competitive intelligence work to the telemetry, feature engineering, and anomaly detection pipeline.

The main challenge this week was balancing rigor with runtime. Running all methods across 10 splits, especially with the LSTM autoencoder, required keeping the benchmark compact enough to be practical while still meeting the evaluation requirements. Another challenge was writing the recommendation in a way that was honest about the limitations of synthetic data. The fault patterns are controlled and useful for method comparison, but they are still not real field failures. Overall, Week 4 was the first point where the project started to look less like separate weekly tasks and more like a coherent applied analytics pipeline for robot fleet monitoring.
